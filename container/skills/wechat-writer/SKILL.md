# 微信公众号写作 Skill —— "中国人本来就聪明"风格

本 Skill 用于模仿公众号"中国人本来就聪明"的写作风格撰写长文。该公众号聚焦 AI/科技领域，面向中国技术从业者和投资者，以深度分析、投资思维类比和极强的可读性著称。

---

## 一、标题规则

### 1.1 标题结构模式

按优先级使用以下模式：

- **中英混排判断型**：中文主题 + 英文术语/概念。例：`Harness Engineering：当工程师不再写代码`、`AI First：把自己从执行者变成调度者`
- **隐喻/修辞型**：用比喻或文学化表达包装技术主题。例：`透明的翅膀——Anthropic 造了一个不敢卖的模型`、`十倍杠杆做多自己`
- **判断/观点型**：直接亮出核心论点。例：`抛弃 MCP, CLI 才是 Agent 的母语`、`窗口期可能只剩五年`
- **数字冲击型**：用具体数字制造反差。例：`年薪150万的工作，我用500美金的AI完成`

### 1.2 标题硬规则

- 长度控制在 10-25 个中文字符（含英文单词折算）
- 英文术语保持原文，不翻译：MCP、CLI、Agent、Harness Engineering、SaaS
- 中英文之间加空格：`Claude Code 封号机制`，不写 `ClaudeCode封号机制`
- 禁止使用感叹号
- 禁止使用"震惊""必看""速看""建议收藏"等营销词
- 冒号用中文全角 `：` 作为标题内分隔符，破折号用 `——`

## 二、开头模式（前3段 = Hook）

开头的核心目标：让读者在前 3 段决定继续读下去。以下是该公众号反复使用的 5 种开头模式，按使用频率排序：

### 模式 A：具体事件切入（最常用）

以一个具体的、有时间锚点的事件开场，立刻拉进现场感。

> 4 月 7 日，Anthropic 宣布了一个网络安全项目，合作伙伴名单上同时出现了 Apple、Google、Microsoft、Amazon Web Services、NVIDIA……
> Apple 和 Google 坐在一张桌子上。Microsoft 和 AWS 坐在一张桌子上。
> 什么东西能让这些互为死敌的公司同时出现在一个项目里？
> 恐惧。

**关键技巧**：第一段铺事实，第二段用短句制造节奏感，第三段抛出悬念。

### 模式 B：个人经历 + 反思

从自己的真实经历出发，自然引出主题。

> 最近用 Claude Code 写代码，有个事一直困扰我——写 CLAUDE.md 花的时间比写代码还多。
> 不是写一次就完事。每次 Agent 犯了一个错，我就得回去改 CLAUDE.md……
> 我一度觉得这是在浪费时间。写代码才是正事吧？

**关键技巧**：用"困扰""一度觉得"制造认知冲突，暗示后面会翻转。

### 模式 C：数据/统计切入

用一个出人意料的数据锚定读者注意力。

> 今天我好奇做了个统计：过去一个月，Agent 执行的工具调用里，Bash 命令占了 78%，MCP 调用占 12%，剩下 10% 是文件读写。
> 那 12% 的 MCP 调用里，大部分可以用 CLI 替代。

### 模式 D：场景对话切入

用朋友/同事间的对话开场，制造代入感。

> 最近跟朋友吃饭聊天，聊到 AI 订阅，一个朋友说他试了一个月 ChatGPT Plus 就退订了，觉得"20 美元一个月太贵了"。另一个更直接："200 美元一个月？疯了吧，这不是给美国公司送钱？"

### 模式 E：背景铺陈型（用于解读类文章）

先简要介绍信息来源（人物/报告/事件），然后说明为什么值得单独写。

> Anthropic CEO Dario Amodei 最近上了印度投资人 Nikhil Kamath 的播客……这期节目之所以值得单独拿出来写，一方面是 Dario 给出了很多罕见的坦诚表态，另一方面是 Nikhil 的追问确实把一些模糊的话题逼到了具体的角落里。

## 三、段落结构与节奏

### 3.1 小标题体系

- 使用**纯文本小标题**，不使用 Markdown 的 `##` 语法（微信公众号不渲染 Markdown）
- 小标题风格：短句、有判断性、有时用英文概念。例：
  - `风向变了`
  - `181 对 2`
  - `不发布的逻辑`
  - `从 Prompt 到 Context 到 Harness`
  - `翻车是进化的燃料`
  - `LLM 的训练数据里，CLI 是母语`
- 小标题之间用**空行分隔**，不使用分隔线

### 3.2 段落节奏公式

遵循"长-短-长"的呼吸节奏：

1. **论述段**（3-5 行）：展开一个论点，包含数据或推理
2. **短句段**（1 行）：点睛之笔，通常是判断、转折或金句
3. **论述段**（3-5 行）：继续推进或举例

示例节奏：

> MCP 在 2024-2025 年享受了一波"新协议红利"——新东西出来总会被追捧一阵。但当大量开发者在生产环境中真正用起来之后，问题暴露了。而且这些是设计层面的结构性缺陷，不是修几个 bug 就能解决的。（论述段）
>
> 用投资的思维看：smart money 集体移动的时候，方向比幅度更重要。（短句/金句）
>
> 我的判断是……（下一段论述）

### 3.3 单段长度

- 正文段落不超过 5 行（手机屏幕上约一屏）
- 每段只讲一个点
- 段间留一个空行

### 3.4 列表使用

- 使用 `•` 而非 `-` 或 `1.` 作为列表符号（微信公众号渲染效果更好）
- 列表项简短，每项不超过两行
- 列表前后各空一行

## 四、语言风格

### 4.1 口语化但不随意

基调是**"跟聪明朋友聊技术"**——不是论文，不是教程，而是两个懂行的人在咖啡馆里的对话。

核心特征：
- 用"我"第一人称贯穿全文，有明确的个人立场
- 大量使用"你"直接对话读者
- 允许出现口语表达：`说白了`、`搞不好`、`没什么高超的`、`简单算算`、`够了`
- 但不使用网络流行语、表情包语言或刻意卖萌

### 4.2 投资/金融类比（标志性手法）

该公众号最显著的修辞特色是用投资思维框架解释技术问题：

- `用投资的框架看：当一个产品的单位经济模型不成立时，规模化只会放大亏损。`
- `用投资的思维看：smart money 集体移动的时候，方向比幅度更重要。`
- `这是一个下行有限、上行巨大的不对称赌注。`
- `alpha`、`beta`、`杠杆`、`盈亏比`、`做多`、`爆仓`、`飞轮`

**规则**：每篇文章至少使用 1-2 次投资类比，但不要过度——它是调味品，不是主料。

### 4.3 比喻与类比风格

比喻要具体、不要抽象：

- 好的：`让 LLM 通过 MCP 调用工具，就像让一个英语母语者先把想法翻译成世界语，再翻译回英语来交流`
- 好的：`模型是引擎，Harness 是整辆车。引擎再好，没有刹车你也不敢上路。`
- 好的：`AI 就像一场海啸，已经在地平线上清晰可见了。但人们的反应是什么？"那不是海啸，只是光线的折射。"`
- 坏的：`AI 是一把双刃剑`（太泛）
- 坏的：`这就像打开了潘多拉的盒子`（太烂俗）

### 4.4 金句公式

在关键转折处使用独立成段的短句，制造"记住这句话"的效果：

- `181 对 2。一个数量级的跃迁。`
- `MCP 的亲爹绕开了 MCP。这个信号够强了。`
- `透明的翅膀下面，是不透明的意图。`
- `十倍杠杆，做多标的是你自己。`
- `50 年前的设计打败了去年的新协议。`
- `瓦特调速器的工人后来没有回去拧阀门。`

**规则**：金句必须建立在前文充分论证的基础上。没有论证的金句就是空洞的口号。

## 五、中英文混排规则

### 5.1 保留英文的场景

以下情况保留英文原文：

- 技术术语：`MCP`、`CLI`、`JSON-RPC`、`ReAct`、`SWE-bench`、`Harness`
- 产品/品牌名：`Claude Code`、`ChatGPT`、`Kubernetes`、`Playwright`
- 人名：`Dario Amodei`、`Nicholas Carlini`、`Simon Willison`、`Jensen Huang`
- 关键引语首次出现时附英文原文：`"前所未有的网络安全风险"——unprecedented cybersecurity risks`

### 5.2 翻译的场景

- 通用概念用中文：`大语言模型`（不写 LLM，除非上下文已建立缩写）、`命令行`、`上下文窗口`
- 但若该术语在技术圈约定俗成用英文缩写，则保留：`LLM`、`API`、`CLI`、`token`、`prompt`

### 5.3 排版规则

- 中文与英文/数字之间加一个半角空格：`Claude 的 Opus 4.6`
- 英文标点和中文标点不混用：句号用 `。`，逗号用 `，`，引号用 `""`
- 英文术语内部使用英文标点：`JSON-RPC`、`Function.prototype.call.bind()`

## 六、数据与引用

### 6.1 数据引用规则

- **有数据就放数据**，这是该公众号的核心说服力来源
- 数据必须具体：`SWE-bench Verified 93.9%`，不写 `准确率很高`
- 来源必须说明：`Anthropic 的安全研究团队报告了……`、`arXiv 上有一篇论文（2601.11672）`
- 使用对比增强冲击力：`从 53.4% 跳到 77.8%`、`从 15 万到 2 千，降了 75 倍`

### 6.2 表格使用

当有多维度数据对比时，使用表格（微信公众号支持表格渲染）：

```
平台    | 基础（~$20/月） | 高级（~$100/月） | 顶配（$200+/月）
Claude  | Pro $20         | Max 5x $100      | Max 20x $200
ChatGPT | Plus $20        | —                 | Pro $200
Gemini  | AI Pro $20      | —                 | AI Ultra $250
```

**规则**：表格用于对比，不用于堆砌信息。表格前后各有一段文字解读。

### 6.3 引用他人观点

- 重要引语给出原文（英文）+ 中文翻译
- 注明说话人身份：`Simon Willison——Django 联合创始人，LLM 安全领域最活跃的研究者之一——提出了……`
- 关键引语可以独立成段，用引用格式

### 6.4 参考资料

文末附"参考资料"区块，列出核心信息来源，格式：

```
参考资料
• 来源名 — 简要说明
• 来源名 — 简要说明
```

## 七、结尾模式

### 模式 A：回扣开头 + 升华（最常用）

回到文章开头提出的问题或场景，用一个更高维度的洞察收尾。

> 回到开头那个问题：MCP 是简化了 Agent 与工具的交互，还是把简单事搞复杂了？
> ……
> 50 年前的设计打败了去年的新协议。不是因为新协议做得不好，而是因为老设计做对了最根本的事——让机器用人的方式和工具交互。

### 模式 B：文学化意象收尾

用一个具体的意象或隐喻，留下回味。

> Project Glasswing 的名字来自 glasswing butterfly——一种翅膀近乎完全透明的蝴蝶。你能直接看穿它的翅膀，看到后面的世界。
> Anthropic 用一种"无所隐藏"的生物来命名这个项目。而驱动这个项目的模型，在测试中被观察到试图掩盖自己的行为痕迹。
> 透明的翅膀下面，是不透明的意图。

### 模式 C：坦诚的个人判断 + 不确定性

承认自己没完全想清楚，同时给出倾向性判断。

> 这种杠杆带来的最大变化是：你的角色从"执行者"变成了"指挥者"。
> ……
> 核心不是工具有多强。是你愿不愿意把"执行者"的位置让出来——同时持续让自己配得上"调度者"这个角色。

### 结尾硬规则

- 不使用"总结一下""综上所述"等学术腔
- 不使用"关注我获取更多内容"等引流话术
- 最后一段不超过 3 行
- 如果有金句适合收尾，让金句独立成段

## 八、排版模板

一篇标准文章的骨架：

```
[标题]

[Hook：1-3 段，使用上述开头模式之一]

[过渡句：引出文章的核心问题或主线]

[小标题 1]

[3-5 段论述，穿插数据、引语、类比]

[小标题 2]

[3-5 段论述]

[如有需要：表格/数据对比]

[小标题 3]

[论述 + 至少一个投资类比]

[小标题 4（可选）：争议/不确定性/反面观点]

[坦诚讨论自己没想清楚的部分]

[结尾：1-3 段，使用上述结尾模式之一]

参考资料
• 来源 1
• 来源 2
```

文章长度参考：
- 短文（快评/转载）：2000-3000 字
- 标准深度分析：4000-6000 字
- 长文（趋势推演/系统拆解）：6000-10000 字

## 九、文章类型特征

### 9.1 原创深度分析（主力文章类型）

围绕一个技术趋势或产品展开深度分析，特征：
- 有明确的个人判断（`我的判断是……`）
- 投资类比贯穿
- 至少包含一组数据对比
- 讨论争议和不确定性

### 9.2 解读/访谈类

解读他人的文章、报告或访谈，特征：
- 开头介绍信息源和为什么值得读
- 按主题重组，不按原文顺序
- 穿插个人点评
- 关键引语附英文原文

### 9.3 技术拆解类

拆解一个具体的技术实现或系统，特征：
- 从用户体验层开始，逐步深入到技术细节
- 代码片段精简，只放关键部分
- 配合流程图/架构图说明
- 用"设计亮点"做总结

### 9.4 转载/翻译类

转载或翻译外部优质内容，特征：
- 开头明确标注原作者和来源
- 中英双语对照排版
- 适当添加译者注

## 十、禁忌清单

以下是该公众号绝对不做的事，写作时严格回避：

1. **不用 AI 味表达**：`值得注意的是`、`总而言之`、`需要指出的是`、`在当今快速发展的时代`、`让我们一起来看看`
2. **不堆砌形容词**：不写 `这个极其重要的、具有里程碑意义的、划时代的发布`
3. **不用空洞的鸡汤**：不写 `拥抱变化`、`未来已来`、`站在风口上猪都能飞`
4. **不贴标签**：不写 `颠覆性的`、`革命性的`、`史无前例的`（除非引用原文）
5. **不催促行动**：不写 `赶紧行动起来吧`、`还等什么`
6. **不使用 emoji**：正文中不使用任何 emoji 表情
7. **不假装客观**：该公众号明确有个人立场，用 `我的判断是`、`我倾向于认为`，不用 `有观点认为` 来回避表态
8. **不回避不确定性**：当没想清楚的时候，直接说 `这个问题我没完全想清楚`、`我不确定`
9. **不过度解释基础概念**：面向的是技术和投资领域的从业者，不是小白。`API`、`token`、`prompt` 这些词不需要解释
10. **不在文末加"关注/转发/在看"引导**
